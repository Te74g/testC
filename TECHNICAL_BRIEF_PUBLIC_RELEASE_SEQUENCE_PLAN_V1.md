# Technical Brief Public Release Sequence Plan V1

## 1. Purpose

This document defines the recommended release order for the public technical brief asset stack.

It exists to answer:

- what should happen after the first public asset decision is made?

## 2. Current Candidate Set

Current ready-but-gated assets:

- `TECHNICAL_BRIEF_SERVICE_PAGE_FINAL_VERSION_DRAFT_V1.md`
- `PUBLIC_TECHNICAL_BRIEF_SERVICE_SUMMARY_FINAL_VERSION_V1.md`
- `PUBLIC_TECHNICAL_BRIEF_PROFILE_FINAL_VERSION_V1.md`
- `TECHNICAL_BRIEF_EXTERNAL_POST_FINAL_VERSION_V1.md`

## 3. Default Recommended Release Order

Default order:

1. service page
2. summary
3. profile
4. post

## 4. Why This Order Exists

Recommended logic:

- the service page is the clearest standalone first-touch asset
- the summary is the best short companion once the core page exists
- the profile becomes more useful after the service line already has a public home
- the post works best as a signal that points toward stronger public assets

## 5. Primary Path

If the sponsor chooses the service page as the first released asset:

### Step 1

Use:

- `TECHNICAL_BRIEF_SERVICE_PAGE_PUBLICATION_DECISION_SHEET_V1.md`

to decide `publish now` or `keep gated`.

### Step 2

If the service page is published:

- move the summary into the active next-release slot

### Step 3

After the summary is either published or intentionally kept gated:

- keep profile as the next reusable support asset

### Step 4

Use the post last:

- as a signal asset
- as a pointer to the stronger public assets

## 6. Fallback Paths

If the service page stays gated:

- keep it as the default first-release candidate
- treat summary as the best fallback release candidate

If the sponsor wants the shortest safe asset first:

- summary may go first
- service page remains the fuller second release candidate

If the sponsor wants the fastest outward signal first:

- post may go first
- but the service page should be prepared as the immediate follow-on destination

## 7. Risk Notes By Order

Service page first:

- best clarity
- moderate public commitment

Summary first:

- easiest safe release
- weaker as a complete standalone offer

Profile first:

- good for marketplace surfaces
- weaker as the main first public explanation

Post first:

- fastest external signal
- highest risk of underspecifying the offer

## 8. Recommended Sponsor Interpretation

Use this plan like this:

- choose the first released asset once
- keep the other assets ready but gated
- do not improvise a release order each time

## 9. Recommended Next Step

First choose the first released asset in:

- `TECHNICAL_BRIEF_FIRST_PUBLIC_ASSET_DECISION_SHEET_V1.md`

Then use the chosen asset's own publication decision sheet to decide:

- publish now
- or keep ready but gated
