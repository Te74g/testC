# Technical Brief Revision Policy V1

## 1. Purpose

This document defines how revisions are handled for the first technical brief service line.

It exists to stop three common failures:

- unbounded free rework
- client re-scoping disguised as revision
- vague disagreement about what was actually included

## 2. Default Rule

Each quoted brief or documentation package should include:

- one bounded revision pass for wording, structure, or emphasis

It should not automatically include:

- unlimited revision rounds
- a new question
- a larger scope than was originally quoted
- implementation work hidden inside a document revision

## 3. What Counts As A Normal Included Revision

The following usually count as one included revision pass:

- clarifying wording
- tightening structure
- reordering sections
- making a recommendation more explicit
- adding a small amount of missing explanation that was clearly implied by the original request
- polishing examples that were already in scope

## 4. What Does Not Count As A Normal Included Revision

The following should be treated as re-scope or new work:

- changing the core question
- expanding from one technical choice to several unrelated choices
- turning a brief into implementation planning that was not included
- adding live integration, deployment, or production troubleshooting
- adding new source material that materially changes the work
- asking for a second audience version with different goals when that was not originally quoted

## 5. Revision Decision Rule

When feedback arrives, classify it first:

### Class A: wording and shape revision

Meaning:

- same question
- same deliverable
- clearer presentation requested

Action:

- use the included revision pass

### Class B: evidence or reasoning challenge inside scope

Meaning:

- same question
- client wants stronger support, stronger caveats, or a different recommendation rationale

Action:

- if the request is still bounded, treat it as the included revision
- if it requires materially new research, re-quote or split scope

### Class C: scope drift

Meaning:

- new question
- bigger surface area
- request now looks like implementation or consulting

Action:

- stop calling it a revision
- write a revised scope or new quote

## 6. Internal Handling Steps

1. Restate the client feedback in plain terms.
2. Classify it as A, B, or C.
3. Check whether the original package still fits.
4. Decide:
   - included revision
   - partial re-scope
   - new quote
5. Record that decision before more work is done.

## 7. Time Boundary

Default operational stance:

- included revision should stay small and fast
- if the revision starts to look like another full pass, stop and re-scope

This policy exists to protect delivery quality, not to minimize useful client feedback.

## 8. Sponsor Boundary

The following still need sponsor review:

- changing the external quoted scope
- changing price or schedule materially
- accepting revision work that crosses the original boundary
- promising implementation work after a documentation quote

## 9. Revision Language For External Use

Preferred wording:

- One bounded revision pass is included for wording, structure, and emphasis adjustments.
- If the underlying question changes materially, I would treat that as a revised scope rather than a normal revision.
- If feedback reveals a larger problem than the original package covered, I would recommend splitting the work instead of hiding that expansion inside a revision.

## 10. Internal Failure Signals

Stop and review if:

- the same client keeps changing the decision question
- revision notes are longer than the original scope summary
- the work is drifting toward implementation
- the new request would change package level or price band

## 11. Current Default

For the current service line:

- include one bounded revision pass
- do not include free re-scoping
- do not treat implementation drift as normal revision

## 12. Next Step

The next related business artifact should be either:

- a one-page public service summary
- or a sponsor-approved example quote using the current packages
