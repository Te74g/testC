# Public Technical Brief Service Summary V1

## Clear Technical Briefs For Messy Decisions

I help turn unclear technical questions into short, readable deliverables that people can actually act on.

Typical work includes:

- technical research briefs
- technical decision memos
- API documentation cleanup
- software and developer documentation rewrites

## Good Fit

This service is a good fit when you need:

- a tooling or architecture choice clarified
- a migration decision framed
- an API or library compared
- rough technical documentation cleaned up
- a short written recommendation before implementation starts

## What You Receive

Depending on the scope, the output can include:

- the real question framed clearly
- option comparison
- recommendation with rationale
- tradeoffs
- unresolved risks
- practical next steps

The goal is to give you something clear enough to review, discuss, and use.

## What This Is Not

This is not disguised implementation work.

It does not automatically include:

- production access
- deployment work
- account setup
- open-ended consulting without scope

If the work expands into implementation, it should be scoped separately.

## Typical Package Shapes

- Fast technical brief
  - best for one narrow technical question
- Decision memo
  - best for larger technical choices with multiple tradeoffs
- Documentation support
  - best for API docs, onboarding docs, or internal developer docs that need structure and cleanup

## Working Style

The style is:

- low-hype
- technically direct
- bounded
- reviewable

If the evidence is weak, I would rather say that clearly than hide it behind polished language.

## Starting Point

If useful, start with a small scoped brief first.

That gives you:

- a concrete output
- a bounded cost
- a clear decision artifact

before committing to anything larger.
