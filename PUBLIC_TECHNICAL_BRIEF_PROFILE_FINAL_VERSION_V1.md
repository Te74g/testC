# Technical briefs, decision memos, and documentation cleanup for messy software questions

I help turn fuzzy technical questions into clear written outputs.

That includes:

- technical briefs
- decision memos
- API documentation cleanup
- developer documentation rewrites

## Good Fit

This is a good fit when you need to:

- compare tools, frameworks, or APIs
- frame a migration or architecture choice
- clean up rough API docs
- rewrite internal developer documentation

## Working Style

My style is:

- direct
- low-hype
- bounded
- reviewable

I try to make the recommendation clear, the tradeoffs explicit, and the next step easy to understand.

## Boundaries

I do not hide live implementation work inside a documentation quote.

If the work expands into production implementation, deployment, or open-ended consulting, that should be scoped separately.

## Starting Point

A good first step is one small scoped brief.

That usually creates:
