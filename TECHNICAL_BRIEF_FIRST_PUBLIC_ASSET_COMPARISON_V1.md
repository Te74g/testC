# Technical Brief First Public Asset Comparison V1

## 1. Purpose

This document compares the four current ready-but-gated public assets.

It exists to answer one sponsor question cleanly:

- which asset should become the first actually released public asset?

## 2. Candidate Set

Current ready-but-gated candidates:

- `TECHNICAL_BRIEF_SERVICE_PAGE_FINAL_VERSION_DRAFT_V1.md`
- `PUBLIC_TECHNICAL_BRIEF_SERVICE_SUMMARY_FINAL_VERSION_V1.md`
- `PUBLIC_TECHNICAL_BRIEF_PROFILE_FINAL_VERSION_V1.md`
- `TECHNICAL_BRIEF_EXTERNAL_POST_FINAL_VERSION_V1.md`

## 3. Comparison Table

| Candidate | Best Use | Main Strength | Main Weakness | Release Risk | Current Recommendation |
|---|---|---|---|---|---|
| Service page | first full public page block | clearest first-touch explanation | longest asset, so public commitment feels stronger | medium | strongest first release candidate |
| Summary | shortest clean offer description | fastest to read and easiest to reuse | can feel too small to carry the whole offer alone | low-medium | strong second release candidate |
| Profile | marketplace bio or public profile | good for profile surfaces and proposal intros | less useful as the first standalone public signal | medium | keep as supporting public asset |
| Post | short public announcement | easiest outward signal and quickest to place | shortest format, highest risk of being underspecified | medium | keep for after one stable core asset is chosen |

## 4. Why Service Page Is Still First

Recommended reason:

- it explains the offer with enough room to be credible
- it keeps the boundary language visible
- it gives the cleanest first-touch public experience
- it is easier to trim down into smaller assets later than to scale a short post upward

## 5. Why Summary Is Second

Summary is the next best candidate because:

- it is compact
- it is easier to publish safely than a post
- it can follow the service page as a short companion asset

## 6. Why Profile And Post Are Not First

Profile is not the first choice because:

- it works best when there is already a clearer core public explanation elsewhere

Post is not the first choice because:

- the short format creates more risk of ambiguity
- it is better as a signal that points to a stronger core asset

## 7. Recommended Publication Order

Recommended order:

1. service page
2. summary
3. profile
4. post

## 8. Recommended Sponsor Decision

Recommended answer:

- choose the service page as the first actual released public asset
- keep summary as the second likely release candidate
- keep profile and post in ready-but-gated state until the first release feels stable
