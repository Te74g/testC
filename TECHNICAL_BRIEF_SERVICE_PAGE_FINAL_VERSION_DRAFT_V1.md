# Clear technical briefs for messy software decisions

Turn one unclear software question into a short, readable deliverable with a recommendation, tradeoffs, risks, and practical next steps.

## Good fit

This service is a good fit when you need:

- a tooling or framework choice clarified
- a migration decision framed
- an API or library compared
- developer documentation cleaned up
- one short written recommendation before implementation starts

## What you receive

Depending on scope, the deliverable can include:

- the real question framed clearly
- option comparison
- recommendation with rationale
- tradeoffs
- unresolved risks
- practical next steps

## Typical engagements

### Fast Technical Brief

Best for one narrow technical question.

### Decision Memo

Best for a larger technical choice with visible tradeoffs.

### Documentation Support

Best for API docs, onboarding docs, or internal developer docs that need structure and cleanup.

## Working style

The style is:

- low-hype
- technically direct
- bounded
- reviewable

If the evidence is weak, I would rather say that clearly than hide it behind polished language.

## What this is not

This service does not automatically include:

- production access
- deployment work
- account setup
- open-ended consulting without scope

If the work expands into implementation, it should be scoped separately.

## Best starting point

The best first engagement is usually one small scoped brief.

That gives you:

- a concrete output
- a bounded cost
- a clear decision artifact

before anything larger is discussed.

## Start here

To scope a first brief, send:

- the question
- the intended reader
- any relevant constraints

That is usually enough to define one bounded first engagement.
