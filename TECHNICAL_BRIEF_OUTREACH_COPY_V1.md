# Technical Brief Outreach Copy V1

## 1. Purpose

This document contains first-pass external-facing copy for the technical brief service line.

It is not auto-approved for publication.

It exists so the sponsor can review real wording before any marketplace profile, DM, email, or proposal is sent.

## 2. Positioning

The core pitch is:

- clear technical thinking
- bounded scope
- readable outputs
- no fake "autonomous AI agency" claims

The tone should feel:

- competent
- direct
- low-hype
- useful to a technical buyer

## 3. Short Service Summary

We turn messy technical questions into clear written answers.

Typical outputs include:

- technical research briefs
- technical decision memos
- API documentation cleanup
- software documentation rewrites

The value is not "AI magic."

The value is faster clarity, clearer tradeoffs, and deliverables that are easy to read and act on.

## 4. Marketplace Profile Blurb

I help clients turn unclear technical decisions into clean, usable documents.

That includes technical briefs, decision memos, API documentation, and software documentation cleanup. My focus is bounded, reviewable work: clear recommendations, stated tradeoffs, unresolved risks, and practical next steps.

If you need production implementation, I can help frame it, but I do not hide implementation work inside a documentation quote.

## 5. Short DM Version

If you have a technical question that is still fuzzy, I can turn it into a brief with:

- the real decision framed clearly
- option comparison
- recommendation
- risks and next steps

This works well for API/tooling choices, migration planning, and documentation cleanup before implementation starts.

## 6. Email-Style Outreach Version

Subject idea:

- Technical brief support for architecture, tooling, and documentation decisions

Body:

I help teams turn messy technical questions into clear written decisions.

Typical work includes short technical briefs, decision memos, API documentation cleanup, and internal documentation rewrites. The goal is simple: make the tradeoffs readable, make the recommendation explicit, and make the next step obvious.

This is a good fit when you need clarity before implementation, or when your documentation exists but is too rough to be useful.

If helpful, I can start with a small scoped brief before any larger engagement.

## 7. Proposal Opener Version

Your request looks like a good fit for a bounded technical brief or decision memo.

My first goal would be to restate the actual decision clearly, compare the viable options, and give you a recommendation with explicit tradeoffs and unresolved risks. That keeps the output useful even if you later choose to implement the work somewhere else.

If you want, I can scope this first as a small deliverable before expanding further.

## 8. Why Me Version

I focus on technical clarity, not fluff.

That means:

- clear framing
- explicit tradeoffs
- readable outputs
- bounded scope
- no pretending documentation is implementation

If the right answer is "do not do this yet" or "the evidence is still weak," I would rather say that clearly than hide it behind nice wording.

## 9. Soft Boundary Language

Use lines like these when needed:

- This quote covers documentation and technical decision support, not live implementation.
- If the work expands into production access or direct integration, I would split that into a separate scope.
- If the inputs are too incomplete for a safe recommendation, I would rather narrow the question first than give you a polished but weak answer.

## 10. Best Initial Use Cases

Use this copy when the client needs:

- an architecture or tooling choice clarified
- a migration decision framed
- API docs cleaned up
- onboarding or developer docs rewritten
- a technical question turned into a short decision memo

Avoid using this copy when the client really wants:

- full implementation
- live deployment
- continuous consulting without scope
- legal, tax, or compliance advice
