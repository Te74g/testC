# Process Order

## 1. Purpose

This document defines the required build order for the two-company AI organization.

The goal is to prevent the project from expanding sideways before the previous layer is stable.

## 2. Core Rule

Build in layers.

Do not widen the system just because a later idea is good.

The default order is:

1. memory
2. governance
3. communication
4. worker bootstrap
5. execution sandbox
6. monetization pilot
7. expansion

If a later phase must start early, record the reason explicitly.

## 3. Phase 0: Memory

Purpose:

- preserve intent through context compression
- preserve current state between work cycles

Required outputs:

- `memory/MEMORY_SYSTEM.md`
- `memory/DURABLE_MEMORY.md`
- `memory/ACTIVE_CONTEXT.md`

Exit criteria:

- intent file exists
- working memory exists
- update rules are mandatory, not optional

## 4. Phase 1: Governance

Purpose:

- define authority
- define approval boundaries
- define company roles

Required outputs:

- `COMPANY_FOUNDATION.md`
- `ORGANIZATION_REGISTRY.md`
- `SPONSOR_APPROVAL_MANUAL.md`

Exit criteria:

- sponsor authority is explicit
- company roles are explicit
- money and irreversible actions are approval-bound

## 5. Phase 2: Communication

Purpose:

- let the companies coordinate without chaos
- make calls, meetings, and brokered research auditable

Required outputs:

- `INTERCOMPANY_PROTOCOL.md`
- `GPT_RESEARCH_BROKER.md`
- shared meeting and approval templates

Exit criteria:

- companies can call each other
- sponsor-routed meetings are defined
- GPT or outside research can be requested in a standard way

## 6. Phase 3: Worker Bootstrap

Purpose:

- define the first small set of usable workers
- avoid creating an uncontrolled worker sprawl

Required outputs:

- first worker roster
- per-worker scopes
- per-worker escalation rules
- per-worker tool boundaries

Exit criteria:

- each worker can be described in one sentence
- each worker has a bounded role
- each worker has a clear escalation target

## 7. Phase 4: Execution Sandbox

Purpose:

- run safe real work before money or high-risk automation is involved

Required outputs:

- local-only execution path
- verification loop
- logging of actions and results

Exit criteria:

- at least one end-to-end task can be completed safely
- the result can be reviewed
- the result can be repeated

## 8. Phase 5: Monetization Pilot

Purpose:

- test a small approved way to create value

Required outputs:

- allowed money-making task categories
- sponsor approval thresholds
- tracking for cost and benefit

Exit criteria:

- at least one approved earning loop exists
- cost is measured
- benefit is measured
- approval rules are preserved

## 9. Phase 6: Expansion

Purpose:

- scale only after evidence exists

Possible outputs:

- more workers
- more hardware
- more API budget
- more specialized operating flows

Exit criteria:

- prior phase results justify expansion
- lessons are written into durable memory
- new scope does not weaken governance

## 10. Default Next Step Rule

When deciding what to do next, first ask:

- which phase are we in
- what is missing from this phase
- what is the smallest document or artifact that unlocks the next phase

This keeps the project from becoming a pile of partially-related ideas.

## 11. Current Status

Current assessed status:

- Phase 0: established
- Phase 1: established
- Phase 2: established at the protocol level
- Phase 3: not yet established
- Phase 4 and beyond: not yet started

## 12. Immediate Follow-Up

The current priority should be:

1. create the first presidents' meeting pack
2. define the first worker roster
3. define the first safe execution loop

That is the shortest path from policy to action.
