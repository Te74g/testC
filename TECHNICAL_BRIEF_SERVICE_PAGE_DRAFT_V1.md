# Technical Brief Service Page Draft V1

## 1. Purpose

This document is a first service-page draft for the technical brief revenue line.

It is meant for:

- simple service pages
- portfolio service sections
- public offer landing-page copy

It is still sponsor-review material, not auto-approved public copy.

## 2. Headline

Clear technical briefs for messy software decisions

## 3. Subheadline

I help turn unclear software questions into short, readable deliverables with a recommendation, tradeoffs, risks, and practical next steps.

## 4. What This Service Is

This service is for situations where the real need is clarity before implementation.

Typical work includes:

- technical research briefs
- technical decision memos
- API documentation cleanup
- software and developer documentation rewrites

The point is not vague consulting language.

The point is to give you something you can actually review, discuss, and use.

## 5. Good Fit

This service is a good fit when you need:

- a tooling or framework choice clarified
- a migration decision framed
- an API or library compared
- developer documentation cleaned up
- a short written recommendation before build work starts

## 6. What You Receive

Depending on the scope, the output can include:

- the real question framed clearly
- option comparison
- recommendation with rationale
- tradeoffs
- unresolved risks
- practical next steps

## 7. Typical Package Shapes

### Fast Technical Brief

Best for one narrow technical question.

### Decision Memo

Best for a larger technical choice with multiple tradeoffs.

### Documentation Support

Best for API docs, onboarding docs, or internal developer docs that need structure and cleanup.

## 8. Working Style

The style is:

- low-hype
- technically direct
- bounded
- reviewable

If the evidence is weak, I would rather say that clearly than hide it behind polished language.

## 9. What This Is Not

This is not disguised implementation work.

It does not automatically include:

- production access
- deployment work
- account setup
- open-ended consulting without scope

If the work expands into implementation, it should be scoped separately.

## 10. Best Starting Point

The best first engagement is usually one small scoped brief.

That gives you:

- a concrete output
- a bounded cost
- a clear decision artifact

before anything larger is discussed.

## 11. CTA

If you have one technical question that is still fuzzy, start by sending:

- the question
- the intended reader
- any relevant constraints

That is usually enough to scope a first small brief.
