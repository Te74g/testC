# Technical Brief External Post Release Playbook V1

## 1. Purpose

This document describes what to do after the sponsor reaches a publication decision on the exact post asset.

It exists so both outcomes stay clean:

- publish now
- keep ready but gated

## 2. Asset In Scope

Primary asset:

- `TECHNICAL_BRIEF_EXTERNAL_POST_FINAL_VERSION_V1.md`

Decision source:

- `TECHNICAL_BRIEF_EXTERNAL_POST_PUBLICATION_DECISION_SHEET_V1.md`

## 3. Path A: Publish Now

Use this path only if the sponsor explicitly approves immediate publication.

### Step 1

Confirm the exact wording is still:

- the same version reviewed
- within the allowed edit boundary
- still aligned with the readiness check

### Step 2

Record the publication status as:

- approved for publication

### Step 3

Use the post as the first exact public post asset for:

- X posts
- LinkedIn-style short updates
- short marketplace announcements
- lightweight public offer signaling

### Step 4

Do not expand claims during formatting work.

Allowed:

- shortening
- line-break changes
- punctuation cleanup
- platform-length trimming
- light wording cleanup

Not allowed:

- new service claims
- broader scope language
- implementation promises
- stronger autonomy language

### Step 5

If the published version is later edited materially:

- return to sponsor review before reuse

## 4. Path B: Keep Ready But Gated

Use this path if the sponsor approves the asset as the standing candidate but does not want publication yet.

### Step 1

Record the publication status as:

- ready but gated

### Step 2

Keep this exact asset as the default exact public post candidate.

### Step 3

Do not rebuild the asset unless one of these becomes true:

- the service line changes materially
- runtime reality changes materially
- the sponsor wants a different post posture or CTA shape

### Step 4

When publication timing changes:

- revisit the decision sheet
- confirm no new claim drift has appeared
- publish only after explicit sponsor release

## 5. Path C: Defer Or Reject

Use this path if the sponsor defers or rejects the current version.

### Step 1

Treat the exact version as:

- not approved for publication

### Step 2

Revise only the areas named by the sponsor.

### Step 3

Return to:

- `TECHNICAL_BRIEF_EXTERNAL_POST_FINAL_REVIEW_SHEET_V1.md`

before any new publication decision.

## 6. Recommended Default

Current recommended default:

- ready but gated

Reason:

- the wording is ready enough
- the remaining uncertainty is publication timing, not wording quality

## 7. Operating Reminder

Publication of this asset should not change the public truth about the organization.

Keep public language aligned with:

- bounded technical brief work
- sponsor-controlled release
- no autonomous unsupervised client delivery claim
